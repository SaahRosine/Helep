The system you are deploying (1-paragraph summary) 
HELEP receives an SOS from a citizen, dispatches the nearest free responder, notifies 
both parties, and feeds a police analytics view with incident counts per zone. All this 
happens via events on Apache Kafka (single broker for local dev, Strimzi-managed in 
K8s) using topics + consumer groups, partitioned by `incident_id` so all events for one 
SOS stay ordered on a single partition. Five services are involved: user-service, sos-
service, dispatch-service, notification-service, analytics-service. The saga 
is choreographed вЂ” there is no central orchestrator. 
Detailed event flow, ports, and SRS mapping are in your own Lecture 4(L4) design-
process doc (you derive it; the lecturer reference is architecture-overview.md). i.e 
use the teachers document to come up with your equivalent architecture-overview 
following the architectural design process. 
Practical Exercise вЂ” Deliverables 
Part A вЂ” Containerization (10%) 
For each of the 5 services, author a Dockerfile that: 
п‚· Uses a multi-stage build (final image based on python:3.12-slim or python:3.12-
alpine). 
п‚· Runs the app as a non-root user (UID в‰Ґ 10001). 
п‚· Exposes only the service port (8001вЂ“8005). 
п‚· Copies only app/ and requirements.txt into the final stage. 
п‚· Passes trivy image --severity HIGH,CRITICAL with zero findings (a clean 
baseline image is acceptable; document any whitelisted CVEs). 
Push images to your own Docker Hub namespace (e.g. dockerhub.io/<you>/helep-
<svc>:0.1.0). 
Part B вЂ” Helm Umbrella Chart (20%) 
Create charts/helep/ as a Helm umbrella chart that contains one sub-chart per 
service. Kafka is managed by Strimzi (Part D) and is not part of this umbrella вЂ” declare it 
as a runtime dependency only (your services reference the bootstrap service <kafka-
cluster>-kafka-bootstrap.kafka.svc.cluster.local:9092). 
Minimum templates required per service sub-chart: 
п‚· Deployment (with livenessProbe + readinessProbe pointing at /healthz + 
/readyz) 
п‚· Service (ClusterIP) 
п‚· ConfigMap for non-secret env (KAFKA_BOOTSTRAP, MATCHER, вЂ¦) 
п‚· Secret for JWT_SECRET 
п‚· HorizontalPodAutoscaler (CPU target 70 %, min 1 / max 3) 
п‚· PersistentVolumeClaim for /data (so SQLite files survive pod restarts) 
Plus an umbrella Ingress exposing user-service, sos-service, and analytics-
service (e.g. on helep.local). 
The values file (values.yaml) must parameterise: image repo + tag, replica count, 
ingress host, Kafka bootstrap server, and MATCHER strategy. 
Part C вЂ” Kubernetes manifest extras (10%) 
Beyond what Helm generates, deliver: 
п‚· A `NetworkPolicy` per service with default-deny ingress + explicit allow for Ingress 
controller, Prometheus scraper, and Kafka brokers in the kafka namespace. 
п‚· A dedicated `Namespace` helep (everything except observability lives here). 
п‚· An `Ingress` resource (NGINX or Traefik вЂ” your choice) routing /api/users/*, 
/api/sos/*, /api/stats/* to the right backends. 
п‚· A `StorageClass` declaration (you can rely on the cluster default, but cite which class 
is used and why). 
Part D вЂ” Kafka via Strimzi Operator (5%) 
Install Strimzi (Helm: oci://quay.io/strimzi-helm/strimzi-kafka-operator) into 
a dedicated kafka namespace, then deploy: 
п‚· A Kafka custom resource named helep-kafka (one broker + one controller 
acceptable for the 24 h time frame вЂ” explicitly cite this limitation in your design 
doc; production requires в‰Ґ 3 brokers + в‰Ґ 3 controllers for replication and quorum). 
п‚· Persistent storage via volumeClaimTemplate in the Kafka CR (в‰Ґ 1 Gi). 
п‚· A KafkaTopic CRD per helep event (7 total): 
п‚· user.registered, sos.triggered, sos.cancelled, responder.assigned, 
п‚· responder.confirmed, safety.zone.entered, notification.sent 
п‚· Each topic: partitions: 3, replicas: 1 (raise to 3 if running в‰Ґ 3 brokers).

п‚· All helep services point at KAFKA_BOOTSTRAP=helep-kafka-kafka-
bootstrap.kafka.svc.cluster.local:9092 (or your chosen cluster name).
